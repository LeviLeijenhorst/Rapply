Executive summary
You can meet your constraints (EU-hosted, no Kubernetes, minimal ops, scale-to-zero GPUs, batch ≥30×
realtime, Dutch+English, ~10 users with 2–3 concurrent jobs, ≈€200/month) by splitting the system into two
parts:
A small always-on “control plane” (FastAPI) that handles uploads, auth, job state, retries, and polling, plus a
serverless  GPU  “data  plane” on  Verda  Serverless  Containers for  the  actual  transcription.  Verda’s
containers  are  explicitly  designed  for  pay-per-use  inference  with  scale-to-zero,  request  queue–based
autoscaling, logs/metrics, async requests, and “batch jobs” for long-running workloads. 
Recommended provider/service (primary): - Verda Serverless Containers - Continuous deployment for
short dictations (2–10s to ~2min). - Jobs/Batch Jobs deployment type for long recordings (up to 90min),
because  Verda  itself  recommends  batch  jobs  for  long  inference  durations  (≈>3  minutes)  to  avoid
downscaling problems.  - Use Verda’s persistent scratch disk (default 500GB) to cache model weights
so cold starts don’t re-download huge files; the docs explicitly call this out as a key use case and note that
storage persists even when scaling down to zero. 
Recommended GPU type (start here): - L40S 48GB or RTX 6000 Ada 48GB on Verda Serverless Containers.
Verda publishes on-demand pricing of  $1.01/h for L40S 48GB and  $0.91/h for RTX 6000 Ada 48GB for
1×GPU serverless container compute types. 
- If you must minimize cost first, Verda lists a 24GB “RTX 4500 Ada or similar” compute type at $0.70/h—
but the 48GB cards give more headroom for batching and concurrency without GPU OOMs. 
Recommended Whisper variant (Dutch+English): - Use Whisper “turbo” / large-v3-turbo for your first
production run because OpenAI states: - Turbo is ~8× relative speed vs “large” and needs ~6GB VRAM (and
relative speeds were measured on an A100 for English; OpenAI notes real-world speed varies by language/
hardware). 
- Turbo is an optimized version of large-v3 with minimal degradation in accuracy. 
- Avoid Distil-Whisper for Dutch: Hugging Face’s Distil-Whisper project and model cards state it is English-
only, and they explicitly recommend Whisper Turbo for multilingual speech recognition. 
Recommended inference engine choice (practical): - If you want the quickest, lowest-drama engineering
path on serverless containers, start with a Transformers-based implementation (like Verda’s own Whisper
async tutorial does) and swap to faster-whisper/CT2 only if needed. Verda provides an end-to-end tutorial
deploying openai/whisper-large-v3-turbo behind FastAPI + Uvicorn + Transformers on Verda. 
- If you want maximum throughput,  faster-whisper (CTranslate2) is a proven speed path: it’s “up to 4×
faster than openai/whisper” and supports large batch speedups in their benchmarks. 
- The faster-whisper README includes benchmark examples where batching drastically reduces wall time

Provider landscape and decision criteria
Why Verda is the best fit for your exact constraint set
Your  constraints  are  unusually  specific:  EU-hosted,  no  Kubernetes,  minimal  ops,  no  always-on  GPU,
programmatic “GPU on/off,” but also reliable and consistently fast.
Verda hits more of these simultaneously than most EU providers because: - Verda serverless containers are
explicitly “pay only for compute in active use,” with scale-to-zero when idle and autoscaling. 
- Autoscaling is queue-based, which is exactly what you want for “2–3 concurrent transcriptions typical” and
bursty usage patterns. 
- Verda supports: - Async inference via HTTP headers (Prefer: respond-async, etc.). 
- Batch Jobs that create a “unique replica for each job” and destroy it when done—explicitly recommended
for long inference durations >3 minutes. 
- Persistent storage that survives scaling to zero, enabling model caching for cold start mitigation. 
- Verda provides a public API and an official Python SDK for programmatic lifecycle control. 
The  other  “EU  GPU  clouds”  you  listed  mostly  fall  into:  “good  VM  GPUs,  but  you  manage  scaling,”  or
“managed inference, but contact sales / enterprise gate.”

Architecture blueprint for a reliable, fast, low-ops pipeline
High-level approach
You want “GPU on only when needed” while still guaranteeing consistent speed and reliability. In practice,
with serverless GPU billing in discrete chunks (Verda: 10-minute intervals), the solution is:
Keep business logic + job state on a cheap always-on CPU service (FastAPI control plane).
Treat GPU inference as an external elastic subsystem:
For short dictations: keep one “continuous” deployment that can warm for a few minutes (scale-
down delay) to amortize cold start/billing chunk. 
For long jobs: run each transcription as a batch job replica so it is destroyed immediately after
finishing, avoiding long idle times and avoiding the risk of downscaling mid-job. 
Use pre-signed upload URLs to object storage so the control plane never becomes a bottleneck and
you don’t proxy huge files through your API server .

Reliability model
Idempotency: Use a single job_id (UUID) as:
Your internal job primary key.
The X-Inference-Id value when calling Verda. Verda supports client-provided IDs, including in
batch jobs docs (“tip: to set a custom job id use the header X-Inference-Id: <custom-id>”). 
Retries: Your control plane retries Verda submission on transient 5xx/network errors, but never
creates a new job id—only replays the same id.
Deadlines: For long jobs, set Batch Jobs “deadline” so hung jobs terminate. Verda batch jobs
explicitly include a deadline after which a replica is destroyed. 
Graceful shutdown: Implement SIGTERM handling in the GPU container so Verda downscaling
doesn’t kill in-flight work; Verda documents a 30s SIGTERM grace period and recommends signal
handling. 
Health checks: Implement /health properly; Verda explains why missing health checks can cause
immediate failures.

Verda deployment configuration
Use persistent disk for model caching
Verda Containers provide a default scratch disk (500GB) mounted at  /data, and explicitly recommend
using it to cache model weights; storage persists even when the deployment scales to zero. 
Set environment variables in the deployment: - HF_HOME=/data/hf-cache
-  WHISPER_MODEL_ID=openai/whisper-large-v3-turbo -  WHISPER_DEFAULT_LANGUAGE= (empty
for auto; or choose “nl” in your dictation product flow)
Continuous deployment scaling defaults (dictation)
Verda’s scaling model uses an internal queue; the default scaling is queue-load only, and you can tune
“queue length per replica” sensitivity. 
Suggested initial values: - Min replicas: 0 (scale-to-zero) 
- Max replicas: 3 (your expected concurrency)
- Max concurrent requests per replica: 1 (start conservative; increase only after benchmarking). Verda notes
concurrency defaults to 1 and can go up to 100; batching helps throughput for some workloads. 
- Queue load target: 0.25 - Verda warns queue load must be interpreted with inference duration (they give
explicit examples mapping queue load to expected waiting time). 
- Scale-up delay: 0s - Scale-down delay: 600s (10 minutes) - This “warm window” is intentionally aligned with
Verda’s 10-minute billing granularity for running replicas. 
- Request message TTL: 180s (dictation should finish fast; fail loudly if not)
Batch Jobs deployment defaults (long recordings)
Verda batch jobs are designed for long-running jobs; they explain why continuous deployments are risky/
costly for long inference (>3 min) and state batch jobs ensure each job gets its own replica destroyed when
finished. 
Suggested initial values: - Max concurrent jobs: 3 (or 2 if you want strict budget control) - Deadline: 1800s
(30 minutes)
- Your target for 90 min @ 30× is ~3 minutes, but deadlines should include fat safety margins for worst-case
audio and cold starts. - Container requirement: process must exit when done; Verda explicitly requires exit
code 0/≠0 to mark success/failure. 
-  In  practice,  you  can  reuse  the  same  server  code  but  add  a  BATCH_MODE=1 env  var  and  call
os._exit(0) after returning the response (use FastAPI BackgroundTasks or similar).

Performance engineering to hit ≥30× realtime consistently
What “≥30× realtime” actually means operationally
Throughput target: - Processing time ≤ audio_duration / 30
Examples: - 2 minutes (120s) dictation must finish in ≤ 4s - 10 minutes must finish in ≤ 20s - 90 minutes
must finish in ≤ 3 minutes
This is demanding for end-to-end (download + decode + model + chunk glue). The speedups come from: -
Choosing turbo (OpenAI reports turbo is ~8× relative speed vs large, with lower VRAM needs). 
- Avoiding cold-start downloads using persistent disk caching (HF_HOME on Verda /data). 
- Autoscaling to multiple replicas so concurrency doesn’t destroy latency. 
Cold start mitigation strategy
Cold starts on GPU serverless typically involve: - Provisioning a GPU worker - Starting your container -
Loading model weights into VRAM - Warming kernels
With Verda: - Use the scratch disk to cache HF weights once (HF_HOME=/data/hf-cache). Verda explicitly
says caching model weights is a key reason for persistent storage and explains that downloading large
weights on every cold start is slow and expensive. 
- Use  scale-down delay as a warm window so bursts of dictations don’t each trigger a cold start. Verda
provides “scale-down delay” as an explicit scaling attribute. 
- Align warm windows with Verda’s 10-minute billing intervals. 
Concurrency strategy
Because you only expect 2–3 concurrent transcriptions: - Prefer replica scaling (1 request per replica) over
high concurrency per replica at first. - Configure max replicas to 3. - Use queue-load tuning so scaling
triggers quickly; Verda documents queue-load semantics and warns that queue-load values must be chosen
based on average inference duration. 
Batching strategy: what to do (and what not to do)
Batching has two meanings:
“Multiple concurrent requests per replica” (Verda setting). Verda notes concurrency defaults to 1
and can be increased; they discuss batching benefits primarily for LLMs, but the mechanism is there.
“Audio chunk batching inside your own code” (you process multiple 30s windows at once). This
can be beneficial but adds complexity and VRAM spikes—avoid initially.

Benchmarking plan and scripts
You need to benchmark quickly on your real audio.
Core metric: Real-Time Factor (RTF)
- RTF = processing_seconds / audio_seconds
- Throughput multiplier = 1 / RTF
Minimum target: RTF ≤ 1/30 ≈ 0.0333
Benchmark matrix - Languages: Dutch, English (because OpenAI states performance varies widely by
language). 
- Durations: 10s, 60s, 5min, 30min, 90min
- Concurrency: 1, 2, 3 jobs simultaneously 
How to run - Upload each file to object storage once. - Start transcription jobs (dictation deployment for
short, batch jobs deployment for long). - Measure “submit → result ready” wall time.
Pass/fail rule (simple) - If 90min Dutch conversation < 180s wall time at concurrency=1 and < ~240s at
concurrency=2–3 (allowing modest slowdown), you’re safe. - If dictation fails due to cold starts, increase
scale-down delay and/or introduce a “prewarm” ping schedule.
Cost model and how to stay near €200/month
The hidden cost driver: billing granularity
Verda serverless containers bill in 10-minute intervals while replicas are running. 
This means sparse short dictations can be expensive if each dictation wakes a GPU for a single billed
increment.
Baseline cost math using Verda published prices
Using Verda on-demand prices (examples): - RTX 6000 Ada 48GB:  $0.91/hr - L40S 48GB:  $1.01/hr - 24GB
“RTX 4500 Ada or similar”: $0.70/hr
Per 10-minute billed chunk: - $0.91/hr → $0.1517 per 10 minutes - $1.01/hr → $0.1683 per 10 minutes -
$0.70/hr → $0.1167 per 10 minutes
So: if your system cold-starts a GPU for a single dictation and then immediately scales to zero, your “per
dictation cost” can look like ~$0.12–$0.17 even if the actual compute took 3 seconds.

Practical cost control levers
Warm window for dictation: keep scale-down delay ~10 minutes so multiple dictations share one

Batch Jobs for long audio: ensures replicas die immediately after job completion; Verda explains

A realistic “€200/month” envelope
At ~$1/hr equivalent: - €200 buys ~200 GPU-hours/month. - If you only run GPUs for ~2 hours/day average,
that’s ~60 hours/month, well under budget.
The bigger risk is not “per-hour GPU price,” it’s “death by warm windows / minimum billed intervals.”
Monitoring, failure handling, and GDPR/security checklist
Monitoring essentials
Verda: - Provides logs and metrics, and advertises “endpoints for Prometheus or Loki” on its serverless
containers page. 
- Also mentions logging and metrics in the console in the containers overview doc. 
Your control plane should expose: -  /metrics Prometheus endpoint (request counts, job latency, error
rates, cold-start proxy metrics) -  /healthz and  /readyz - Structured logs with  job_id,  user_id, 
provider_request_id
Most likely failure modes and fast fixes
Cold start takes too long: - Fix: ensure  HF_HOME points to  /data and your deployment has persistent
scratch disk (default). Verda explicitly recommends this for caching weights. 
- Increase scale-down delay to maintain a warm window. 
New replicas return 500s right after scaling up: - Fix: implement  /health correctly. Verda warns that
without health checks, new containers may receive work before ready and fail; they also note they pass
errors through. 
Long jobs get killed mid-run: - Fix: do long jobs with Batch Jobs deployment type. Verda explicitly created
this to avoid downscaling issues for >3 minute inference. 
- Increase batch job deadline. 
Duplicate transcriptions (double billing): - Fix: set X-Inference-Id to your internal job_id and treat
job submission as idempotent. Verda supports client-provided IDs.
